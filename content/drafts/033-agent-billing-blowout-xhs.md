## 正文

我把这个季度三起企业 AI 账单事故的报道原文对了一遍——Tech Startups 5 月 28 日、Windows Central 5 月 14 日、Fortune 和 Yahoo Finance 5 月 25–26 日。一家匿名企业 30 天烧了 $5 亿，Uber 四个月烧完全年 $34 亿预算，Microsoft 直接取消了整个 Windows/Office 部门的 Claude Code 许可证。三家，同一个季度。

三篇报道的结论都指向"管理层没设好上限"。但三家里有两家设了。Uber 有年度预算框架和审批流程，Microsoft 有部门级别的许可证管理。结果一样：预算击穿。

5 月 28 日那篇最值得细看。一个 AI 顾问披露：客户给员工开放 Claude 无限制访问后，工作负载大规模迁移到 Agent 形态。软件开发 Agent 在自动化循环里运行——并行启动子任务，每个子任务再调用代码执行器、测试框架、文档检索。不停，不等确认。单次任务的 token 消耗量最高是普通对话的 1000 倍。

Uber 的数据更完整。5000 名工程师部署 Claude Code，到 4 月月均使用率 84–95%，每人月均 API 费 $500–$2,000，四个月吃完全年额度。Microsoft 的 Experiences & Devices 部门——管 Windows、Microsoft 365、Outlook、Teams、Surface 的那个——部署 Claude Code 后工程师用量超标，5 月 14 日宣布截至 6 月 30 日取消大多数许可证，转向 GitHub Copilot CLI。

问题出在比"上限"更底层的地方。

所有 enterprise IT 采购合同——每席位授权、月度预算、年度框架——都建在一个假设上：用户以人的速度使用软件。人一个月能在 Claude 里打多少轮对话，有个自然天花板。Agent 没有天花板。Agent 不开会，不等 review，不下班。它在循环里跑，一个 ticket 吃的 token 可能超过一个工程师一个月的手动量。

CTO 团队设计了 Agent 工作流。IT 采购签了合同。Finance 以为这是 SaaS 年费。三段链路，三个部门，没有一个人从头看完。

**这不是管理失误。这是 enterprise IT 的整套定价和预算架构，从未被 Agent 速度测试过。为人的速度设计的框架，被 Agent 的速度击穿了。**

下一批账单已经在路上。因为问题不在用了什么工具——在于企业内部负责为 Agent 工作流建立用量模型的那个角色，目前还不存在。**谁该填这个位置？**
