## 正文

我以前以为 Multi-agent 是过度工程。

100 个 Agent 协作？听起来像分布式系统里"微服务拆太碎"的教训重演。模型够强的时候，一个就够了——把任务丢给 Claude 或 GPT，等结果。为什么要协调 100 个各司其职的 Agent？

5 月 12 日，Microsoft 公开了 MDASH。全称 Multi-model Agentic Scanning Harness。100 多个专门化 AI Agent 协同工作，目标是 Windows 的网络和认证代码栈。结果：自主发现 16 个新漏洞，其中 4 个严重远程代码执行（RCE），涉及 Windows 内核 TCP/IP 栈和 IKEv2 服务。CyberGym 公开安全基准排名第一，88.45%，比第二名 Anthropic Mythos 高出约 5 个百分点。全程无人工介入。漏洞已用于同月 Patch Tuesday 生产补丁。

Anthropic 的 Mythos 是另一条路线：一个前沿大模型扫描代码，靠模型本身的广度覆盖所有漏洞类型。Microsoft 选了相反的架构——不让任何单个 Agent 做整件事。每个 Agent 只看一种漏洞模式。发现者和验证者是不同 Agent。一个说"这里有缺陷"，另一组尝试证明它错了。只有当反驳失败，才进入下一阶段的利用性证明。

辩论，共识，验证。在 clfs.sys 上 96% 召回率，tcpip.sys 上 100%。

为什么 100 个窄 Agent 跑赢了 1 个宽模型？我的理解是任务本身决定了架构选择。安全漏洞不是一类东西。TCP/IP 栈的逻辑漏洞和认证协议的状态机缺陷需要完全不同的推理路径。一个"什么都懂一点"的通用模型在每个子领域的深度都有天花板。100 个各自只精通一件事的 Agent 没有这个天花板——它们用协作弥补广度，用专注换取深度。

而且辩论机制解决了一个关键问题：幻觉。单模型输出一个结论，没有内置校验。MDASH 的 Agent 互相证伪——你说有洞，我来试试能不能打。这是结构性地把幻觉率压下去。

这不只是安全领域的结论。

过去一年做 Agent 产品的默认思路是"等更强的模型出来"。模型能力曲线在涨，所以"用一个更强的模型"永远比"搭一个更复杂的系统"看起来性价比更高。MDASH 给出了一个边界条件：当任务需要的是深度专业判断——不是广度，是深度——多个专门化 Agent 协作能触达单一模型无法到达的精度。

**专门化协作不是退步，是在特定条件下唯一能抵达的精度边界。**

如果你在设计 Agent 系统：你的任务到底需要一个更大的模型，还是一群各自只做一件事的小 Agent？
